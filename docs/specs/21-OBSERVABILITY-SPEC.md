# Spec 21 — Observability Specification

Status: Draft v0.1 · Last updated: 2026-07-18

Plans debugging NovaOS itself before implementation, per Staff Engineer review, pulling
together the logging conventions already stated in
[../01-SYSTEM-ARCHITECTURE.md](../01-SYSTEM-ARCHITECTURE.md) §5 and
[../11-CODING-STANDARDS.md](../11-CODING-STANDARDS.md) §5 with new metrics/tracing/
health-check machinery every service RFC references but none fully define.

## 1. Structured Logging

- Every Nova process writes structured, leveled log lines
  (`error`/`warn`/`info`/`debug`/`trace`) to `/nova/logs/<service>.log`
  ([19-FILESYSTEM-LAYOUT-SPEC.md](19-FILESYSTEM-LAYOUT-SPEC.md) §1), one JSON object per
  line: `{"ts", "level", "service", "message", "fields": {...}}` — machine-parseable by
  Nova Monitor without a custom log-format parser.
- Rotation: size-based (10MB per file, 3 files retained per service) — no time-based
  rotation, since a chatty service under a debug session shouldn't have its history cut
  off by a wall-clock boundary; a quiet service's 10MB might span weeks, which is fine.
- Log level is runtime-adjustable per service via `nova.debug.set_log_level {service,
  level}` (a privileged Nova Bus call, restricted to Nova Monitor's Developer Mode,
  mirroring [09-APPLICATION-SPECS.md](09-APPLICATION-SPECS.md) "Nova Terminal"'s
  Developer Mode escape hatch) — no service restart required to turn up verbosity while
  chasing a bug.
- Per-service "never log" rules are enumerated in that service's own RFC (§Logging
  section, e.g. [RFC-0002](../rfcs/RFC-0002-nova-bus.md) never logs payload content,
  [RFC-0005](../rfcs/RFC-0005-notification-service.md) never logs notification bodies)
  — this document doesn't repeat them, it's the place that *requires* every RFC to have
  one.

## 2. Metrics

- Every resident service ([../01-SYSTEM-ARCHITECTURE.md](../01-SYSTEM-ARCHITECTURE.md)
  §3) publishes a periodic (5s interval) snapshot on `nova.monitor.metrics.<service>` —
  a fixed, small set of counters/gauges defined in that service's own RFC (§Metrics
  section).
- Metric types: `Counter` (monotonic, e.g. "messages routed"), `Gauge` (point-in-time,
  e.g. "current RSS"), `Histogram` (distribution, e.g. frame time — reported as
  precomputed p50/p99 rather than raw samples, to keep the metrics payload small and
  avoid the observability system itself becoming a performance cost worth measuring).
- Nova Monitor subscribes to `nova.monitor.metrics.*` (wildcard,
  [15-NOVA-BUS-PROTOCOL-SPEC.md](15-NOVA-BUS-PROTOCOL-SPEC.md) §7) and is the only
  first-party consumer in v1 — no separate metrics-storage daemon, no time-series
  database; Nova Monitor holds a bounded in-memory ring (last N samples per metric,
  matching the boot-ring-buffer pattern in
  [03-BOOT-TIMELINE.md](03-BOOT-TIMELINE.md) §4) for its own charts, discarding older
  samples rather than persisting a full history — consistent with "no unnecessary
  daemon" applied to observability infrastructure itself.

## 3. Tracing (Cross-Service Correlation)

- Every `Call` in the Nova Bus protocol already carries a `correlation_id`
  ([15-NOVA-BUS-PROTOCOL-SPEC.md](15-NOVA-BUS-PROTOCOL-SPEC.md) §5) scoped to a single
  request/response pair — tracing extends this with a **trace_id**, a separate field
  threaded through an entire multi-hop flow (e.g., the app-launch flow spanning
  `nova-shell → nova-sessiond → permission-broker → nova-compositor`,
  [01-INTERACTION-FLOWS.md](01-INTERACTION-FLOWS.md) §1).
- A `trace_id` is generated by the flow's originating call (e.g., `nova-shell` when a
  user clicks an app icon) and propagated by every service that makes a further Bus
  call as part of handling that request — carried as an optional `Envelope` field
  (added to the protocol in [15-NOVA-BUS-PROTOCOL-SPEC.md](15-NOVA-BUS-PROTOCOL-SPEC.md)
  §3's `Envelope` message as `optional uint64 trace_id = 13`), not a payload field, so
  the broker itself can log it without deserializing opaque payload bytes.
- Every log line emitted while handling a traced call includes the `trace_id` — Nova
  Monitor's Developer Mode can filter logs across all services down to a single
  end-to-end flow, which is what makes "why did this app launch take 800ms" answerable
  without manually correlating timestamps across five separate log files.
- Tracing is **opt-in per-flow, not always-on**: only flows explicitly instrumented at
  their origin generate a `trace_id` (app launch, package install, OS update — the
  handful of flows in [01-INTERACTION-FLOWS.md](01-INTERACTION-FLOWS.md) that are
  multi-hop and latency-sensitive enough to warrant it); simple single-hop calls never
  pay the (small) overhead of trace propagation.

## 4. Debug Overlays

- `nova-compositor` supports a compile-time-gated (debug builds only — never present in
  a release image, per [22-RELEASE-PROCESS-SPEC.md](22-RELEASE-PROCESS-SPEC.md) §1) debug
  overlay, toggled by a keybinding: shows per-output frame time, damage region
  rectangles drawn as outlines over the actual desktop content, and current z-layer
  ([04-WINDOW-MANAGER-SPEC.md](04-WINDOW-MANAGER-SPEC.md) §5) for the window under the
  cursor.
- This directly visualizes the damage-tracking model in
  [04-WINDOW-MANAGER-SPEC.md](04-WINDOW-MANAGER-SPEC.md) §7 — the primary intended use
  is diagnosing "why is this animation janky" or "why is this idle app still causing
  repaints" during development, not a production feature.

## 5. Health Checks

- Every resident service exposes liveness the same way: presence on the Nova Bus (a
  connected socket, [15-NOVA-BUS-PROTOCOL-SPEC.md](15-NOVA-BUS-PROTOCOL-SPEC.md) §12's
  keepalive) *is* the health signal — there is no separate `nova.health.check` request/
  response call every service must implement. `nova-sessiond`'s crash-restart policy
  (referenced in every RFC's §Recovery Strategy) already depends on exactly this signal
  (a dead socket connection), so a separate health-check protocol would be redundant
  machinery layered on top of a mechanism that already exists.
- The one place a richer health signal exists is the OS-update post-boot check
  ([16-STATE-MACHINES.md](16-STATE-MACHINES.md) §4's `HealthCheck` state) — "healthy"
  there specifically means `nova-compositor` and `nova-sessiond` both reached `ready`
  within the boot budget ([03-BOOT-TIMELINE.md](03-BOOT-TIMELINE.md)), which is a
  boot-specific, one-time check, not a recurring runtime health-check loop.

## 6. What Nova Monitor Surfaces (consumer-side summary)

Ties §1–§5 together into what a user/developer actually sees, cross-referencing
[09-APPLICATION-SPECS.md](09-APPLICATION-SPECS.md) "Nova Monitor": Overview tab (metrics,
§2), Processes tab (per-app cgroup stats, [RFC-0008](../rfcs/RFC-0008-session-manager.md)),
Boot tab (the boot ring buffer, §5 + [03-BOOT-TIMELINE.md](03-BOOT-TIMELINE.md) §4), and
a Developer Mode-gated Logs/Trace view (§1, §3) not shown in the default UI to keep the
everyday Monitor experience simple for non-developers.
